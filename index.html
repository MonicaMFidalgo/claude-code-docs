Claude Code Memory System

Two complementary systems give Claude persistent context about your project, preferences, and working style — so you never repeat yourself across sessions.

📄 CLAUDE.md

Developer-Written

You write the rules. Project-level, committed to git, always loaded at startup.

written by: you

Claude-Written

written by: Claude

CLAUDE.md Files

Persistent developer-controlled instructions that load automatically at every session start.

CLAUDE.md is a plain Markdown file you create in your project. Claude reads it at the start of every session — before you type anything — giving it persistent context about your codebase, conventions, and expectations.

💡 Think of CLAUDE.md as your team's coding conventions document, permanently injected into Claude's context. Anything you'd normally explain to a new engineer on day one belongs here.

What to put in it

  • ›Build and test commands (npm test, yarn build)
  • ›Project architecture and key directories
  • ›Coding standards and naming conventions
  • ›Patterns to use and patterns to avoid
  • ›Common workflows and gotchas

When does it load?

Project-root CLAUDE.md

Loads at every session start. Survives /compact. Committed to git — shared with your team.

Sub-directory CLAUDE.md

Loads on demand when Claude reads files in that directory. Reloads after /compact as needed.

Claude discovers CLAUDE.md files by walking up the directory tree from your current working directory. All discovered files are concatenated — they don't override each other.

The quality of your CLAUDE.md directly impacts how well Claude follows it. Vague instructions produce inconsistent results; specific, verifiable instructions produce consistent results.

The verifiability test

Before adding any instruction, ask: "Could I verify this with a grep, linter, or test?" If not, it's too vague.

✗ Vague (avoid) 
# Rules
Write clean, readable code.
Use good naming conventions.
Handle errors properly.
Follow best practices.
✓ Specific (use this) 
# Rules
- Max line length: 100 chars (enforced by ESLint)
- All functions must have JSDoc comments
- Use named exports only (no default exports)
- Errors: always use Result<T, E> pattern,
  never throw unless crossing async boundaries
- Test files must use describe/it nesting

Key guidelines

  • ›Keep the file under 200 lines — Claude may truncate larger files silently
  • ›Use markdown headers to organize sections (Build, Test, Conventions, etc.)
  • ›Review periodically — remove outdated or conflicting rules
  • ›Stick to facts Claude needs in every session — don't include ephemeral context
⚠ Claude silently truncates CLAUDE.md beyond its context limit — you won't get an error. Instructions past the threshold simply vanish. Check your line count with wc -l CLAUDE.md.

When your CLAUDE.md grows beyond 200 lines, split it using the @path/to/file import syntax. Place the directive on its own line — both relative and absolute paths work. Imports support up to 5 levels of nesting.

.claude/rules/ directory

For modular, topic-focused rules, place Markdown files in .claude/rules/. Files without a paths: frontmatter load at launch like a regular CLAUDE.md import.

.claude/
├── rules/
│   ├── typescript.md      # TypeScript conventions
│   ├── testing.md         # Test file rules
│   ├── api-design.md      # API patterns
│   └── security.md        # Auth & security rules
├── CLAUDE.md              # Root project rules (uses @imports)
└── CLAUDE.local.md        # Personal overrides (gitignored)
✓ Imports are resolved relative to the file containing the @ directive. So @.claude/rules/testing.md in your project root CLAUDE.md resolves from the project root.

Add YAML frontmatter with a paths: field to a rules file to scope it to specific file patterns. Claude only applies those rules when working with matching files — keeping context focused and relevant.

ℹ Path patterns use glob syntax. Use ** for any directory depth, *.ts for file extensions, and brace expansion like *.{ts,tsx}. 
---
paths:
  - "src/api/**"
  - "src/routes/**"
---
# API Handler Rules

- Validate all request bodies with Zod before processing
- Return `{ success: false, error: string }` on all error paths
- Log every 4xx and 5xx response with `logger.warn()`
- Never expose stack traces or internal error details in responses
- All handlers must be wrapped in the `withAuth` middleware
---
paths:
  - "**/*.test.ts"
  - "**/*.spec.ts"
  - "src/tests/**"
---
# Test File Rules

- Use `describe` + `it` nesting (never bare `test()`)
- Mock all external HTTP calls with `msw` — no real network calls in tests
- Each test file must have `beforeEach(() => jest.clearAllMocks())`
- Use `expect.assertions(N)` in async tests to catch silent misses
- Do NOT mock the database — tests must hit the real test DB
✓ Path-scoped rules prevent context bloat. Your API handler rules won't clutter Claude's context when you're editing a React component.

The /init command scans your project and generates a starter CLAUDE.md automatically. It's the fastest way to bootstrap a new project.

1

Open Claude Code in your project root

cd /path/to/your/project
claude
2

Run the init command

/init
3

Claude scans your codebase

It reads your package.json, detects your framework, and generates a contextual draft covering build commands, test setup, and project structure.

4

Review and customize the generated file

The draft is a starting point — add your team's conventions, patterns to avoid, and architectural decisions.

⚠ Don't commit the /init draft as-is. Review it, trim anything incorrect, and add the team-specific conventions that Claude couldn't infer from code alone.

Auto Memory

Notes Claude writes itself based on your corrections, preferences, and ongoing project context. Requires Claude Code v2.1.59+.

Auto memory lets Claude remember what it learned last session — your corrections, confirmed patterns, active projects — without you having to re-explain every time. Claude decides what's worth saving based on future utility.

CLAUDE.md Auto Memory
Written by Developer Claude
Triggered by Always at startup Corrections, preferences, explicit asks
Committed to repo? Yes (usually) No — machine-local only
Shared with team? Yes No — per-machine, per-user
Survives /compact? Project root does Yes
Human-editable? Yes (it's your file) Yes (via /memory or directly)
✓ Auto memory is additive — it supplements CLAUDE.md, it doesn't replace it. Use CLAUDE.md for team-wide rules, and auto memory for individual preferences and learnings.

Auto memory is organized into four types. Each has a specific purpose — knowing which type to use (or ask Claude to use) makes memories more findable and useful.

Best Practices

Patterns that make the difference between Claude that reliably follows your rules and Claude that doesn't.

The single most important hygiene rule: keep CLAUDE.md under 200 lines. Beyond that, instructions can be silently truncated.

# Check line count
wc -l CLAUDE.md

# Check size
du -h CLAUDE.md

# If over 150 lines, start splitting into .claude/rules/ files

Regular maintenance checklist

  • Line count stays under 200 (ideally under 150)
  • No conflicting rules between CLAUDE.md and imported files
  • Outdated instructions removed (e.g., old test runner, deprecated patterns)
  • CLAUDE.local.md is in .gitignore
  • Run /memory to verify all files are loading
⚠ There's no error when CLAUDE.md is too large. Claude doesn't say "I truncated your file." The instructions just silently disappear from context. Monitor your line count.

Instruction quality determines consistency. Vague rules get vague results.

❌ Vague→✓ Better✓✓ Best (verifiable)
Write clean code → Use consistent naming camelCase for variables, PascalCase for classes, SCREAMING_SNAKE for constants
Handle errors properly → Don't swallow errors All caught errors must be logged with logger.error() before re-throwing or returning
Write good tests → Tests should be readable Each test: one assertion per it(), descriptive test name answering "when X, should Y"
Keep functions small → Avoid long functions Functions over 40 lines must be refactored — split by responsibility, not by length
✓ The gold standard: if a linter, grep, or code review could verify the rule, Claude can apply it consistently. If it requires judgment, it'll be applied inconsistently.

As projects grow, a single CLAUDE.md becomes unwieldy. The .claude/rules/ pattern keeps rules modular and focused — one topic per file.

.claude/
├── rules/
│   ├── typescript.md           # TypeScript-specific rules (always loads)
│   ├── testing.md              # Test patterns (paths: **/*.test.ts)
│   ├── api-handlers.md         # API rules (paths: src/api/**)
│   ├── security.md             # Auth rules (paths: src/auth/**)
│   ├── react-components.md     # UI rules (paths: src/components/**)
│   └── database.md             # DB rules (paths: src/db/**)
├── CLAUDE.md                   # Root rules + @imports (keep < 50 lines)
└── CLAUDE.local.md             # Personal prefs (gitignored)

When you want the same rules to apply across multiple repositories, use symlinks to maintain a single source of truth.

Symlink a shared rules directory

# In your dotfiles repo, create a shared rules directory
mkdir -p ~/dotfiles/claude-rules

# Add shared rules files
echo "Always use named exports" > ~/dotfiles/claude-rules/exports.md

# In each project, symlink the shared directory
cd /path/to/project
ln -s ~/dotfiles/claude-rules .claude/rules/shared

# Or symlink individual files
ln -s ~/dotfiles/claude-rules/typescript.md .claude/rules/team-typescript.md

User-level rules (apply to every project)

# Rules in ~/.claude/rules/ apply to ALL your projects
mkdir -p ~/.claude/rules

# Example: personal style rule that applies everywhere
cat > ~/.claude/rules/my-style.md << 'EOF'
# My Personal Rules
- Show diffs before applying any refactor
- Explain the "why" before the "what" in code reviews
EOF
ℹ Symlinks work for both files and directories. The .claude/rules/ directory supports any depth of organization — you can use subdirectories like .claude/rules/frontend/ and .claude/rules/backend/.

Troubleshooting

Diagnosing and fixing the most common memory system issues.

Claude ignoring your CLAUDE.md is the most common issue. Work through this checklist in order.

Diagnosis checklist

  • Run /memory inside Claude Code — verify your file appears in the loaded files list
  • Check the file is at project root (not nested two levels deep in a subdirectory)
  • Verify line count: wc -l CLAUDE.md — must be under 200
  • Look for conflicting instructions across multiple CLAUDE.md files at different levels
  • Check YAML frontmatter syntax in path-scoped rule files (malformed YAML silently skips the file)
  • For path-scoped rules: confirm you're editing a file that matches the paths: pattern
# Quick YAML syntax check (requires python)
python3 -c "import yaml; yaml.safe_load(open('.claude/rules/api-handlers.md').read())"

# If no output, YAML is valid. If error, fix the frontmatter.
ℹ Use --append-system-prompt to inject instructions at the system prompt level, which takes the highest priority. Useful for debugging whether the issue is loading vs. priority.

Running /compact summarizes the conversation to free up context window space. It behaves differently for different CLAUDE.md file types.

FileSurvives /compact?Why
./CLAUDE.md (project root) ✓ Yes Always reloaded at the start of a compacted session
./src/CLAUDE.md (subdirectory) ~ On demand Reloads when Claude reads a file in that directory
~/.claude/CLAUDE.md (user) ✓ Yes Loaded at every session start, including post-compact
Auto memory (MEMORY.md) ✓ Yes Stored on disk, not in context — reloads fresh
Conversation context ✗ Summarized This is what /compact compresses — not CLAUDE.md files
✓ Fix: If you're losing instructions after /compact, move them from conversation context into your project-root CLAUDE.md. That file always reloads.

Full Project Audit

The inspector above only sees one file. A real project splits configuration across .claude/rules/, .claude/skills/, hooks, and sub-directory CLAUDE.md files. Copy the prompt below and run it directly in Claude Code inside your project — Claude can actually read the full .claude/ directory and give accurate, project-specific feedback.

▶ Open your project in Claude Code, paste the prompt below into the chat, and press Enter. Claude will read every file in .claude/ before responding — no manual copy-pasting required.
Audit prompt — paste into Claude Code 
Read everything in .claude/:
- CLAUDE.md (project root)
- All files in .claude/rules/
- All SKILL.md files in .claude/skills/
- .claude/settings.json (hooks)
- Any sub-directory CLAUDE.md files

Then audit the complete Claude Code configuration and report findings in three groups:

## ✓ Well-configured
List what's already solid and why.

## ⚠ Missing or needs improvement
For each gap, include:
- What's missing
- Why it matters
- A ready-to-paste fix (exact content to add, with file path)

Audit these areas:

**Structure**
- Is CLAUDE.md under 200 lines? Is it lean (overview + @imports only)?
- Are rules split into .claude/rules/ with @imports?
- Are path-scoped rules used where appropriate (YAML `paths:` frontmatter)?
- Are complex multi-step workflows in .claude/skills/ rather than CLAUDE.md?

**Content**
- Are build, test, and lint commands documented somewhere in the config?
- Is there a project overview explaining what the product does and who uses it?
- Is there a "Common Gotchas" section for project-specific traps?
- Are there instructions for proactive test maintenance?
- Are instructions specific and verifiable (not vague like "write clean code")?
- Are there any conflicting instructions ("always X" vs "never X")?

**Automation**
- Does settings.json have PostToolUse hooks for typecheck/lint after file edits?
- Are there any missing hooks for common failure modes in this project?

**Memory alignment**
- Run /memory and check if any auto-memory corrections duplicate CLAUDE.md rules.
- Flag any that should be consolidated into a canonical .claude/rules/style.md.
